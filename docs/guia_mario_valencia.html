Día 1 · Lección 1
¿Qué es un prompt para un programador?
30 min · Hermes Agent en tu estación de trabajo
Concepto clave
Un prompt para desarrollo es como escribir un ticket de tarea bien especificado: lenguaje, contexto, restricciones y output esperado. Cuanto más preciso eres, más útil es lo que obtienes.
🎯
Tu meta es Instructor. Eso significa que vas más rápido que los demás Champions. Esta guía tiene actividades con mayor profundidad técnica para que puedas enseñar lo que aprendas.

Iniciar Hermes para desarrollo

# Opción 1 — consulta única con herramienta de código
hermes chat -t terminal,file -q "tu pregunta de código aquí"

# Opción 2 — sesión interactiva (recomendada para pair programming)
hermes

Práctica — compara estos prompts

Prompt A — Sin contexto:

hermes chat -q "revisa este código y dime si está bien"

Prompt B — Con contexto de desarrollo:

hermes chat -t file -q "Eres un senior developer con experiencia en Java y sistemas universitarios. Revisa el siguiente fragmento de código del módulo de Graduados de la UJMD. Identifica: bugs potenciales, mejoras de legibilidad, y si hay patrones que deberían refactorizarse. Explica cada hallazgo con el porqué. [pega tu código aquí]"

Actividades del Día 1

  • Ejecuté el Prompt A con un fragmento de código real
  • Ejecuté el Prompt B con el mismo fragmento y noté la diferencia
  • Creé mi propio prompt de revisión de código con contexto del área de Desarrollo
  • Guardé ambas respuestas para comparar
💡
Tu experiencia previa: Ya usaste Gemini para completar el módulo de Graduados. Hoy usas Hermes con el mismo tipo de tarea pero con prompts más estructurados.
📎 Evidencia del Día 1
  • Respuesta del Prompt A (sin contexto)
  • Respuesta del Prompt B (con contexto de desarrollo UJMD)
Día 2 · Lección 1 (práctica)
3 prompts de desarrollo que usarás cada semana
45 min · Hermes Agent con toolset de código

Prompt 1 — Code review de código legado

hermes chat -q "Eres un senior Java developer. Analiza el siguiente código legado e identifica: 1) bugs o riesgos, 2) código duplicado que podría abstraerse, 3) mejoras de rendimiento. Para cada hallazgo da una sugerencia concreta de cómo arreglarlo. Código: [pega aquí]"

Prompt 2 — Generación de documentación técnica

hermes chat -q "Genera la documentación técnica en español para la siguiente función/módulo. Incluye: descripción general, parámetros de entrada, valor de retorno, excepciones posibles y ejemplo de uso. Código: [pega aquí]"

Prompt 3 — Levantamiento de requerimientos

hermes chat -q "Eres un analista de sistemas universitarios. A partir de esta descripción de usuario, genera un documento de requerimientos funcionales con: casos de uso, criterios de aceptación y preguntas que debo aclarar con el solicitante. Descripción: [escribe aquí el requerimiento que te llegó]"
⚠️
Tu bloqueo — Acceso y permisos: Si Hermes necesita acceder a archivos locales del proyecto, usa hermes chat -t file -q "...". Si el problema es de permisos a nivel de SO, consulta con Douglas — no intentes saltarte permisos institucionales.

Actividades del Día 2

  • Ejecuté el Prompt 1 con código real del área (sin datos de producción)
  • Ejecuté el Prompt 2 y generé documentación de alguna función existente
  • Ejecuté el Prompt 3 con un requerimiento real reciente
  • Guardé los 3 prompts en un doc para reutilizar
  • Identifiqué cuál de los 3 ahorra más tiempo en mi flujo real
📎 Evidencia del Día 2
  • Los 3 prompts con sus resultados
  • Nota: ¿cuál fue más útil para tu flujo de trabajo?
Día 3 · Lección 2
Pair programming con Hermes — refactorización iterativa
40 min · Hermes Agent modo interactivo
Concepto clave
Hermes como pair programmer: le das el código, te da feedback, le pides que refactorice, le preguntas por qué, le pides alternativas. Es un ciclo continuo — no una sola pregunta.

Sesión de refactorización en modo interactivo

# Inicia sesión
hermes

# Ciclo de refactorización:
> [pega el código que quieres refactorizar]
> ¿Qué problemas identificas en este código?
> Ahora refactorízalo aplicando las mejoras que mencionaste.
> Explica qué cambió y por qué cada cambio es mejor.
> ¿Hay algún patrón de diseño que aplique aquí?
> /quit

Actividades del Día 3

  • Inicié Hermes en modo interactivo
  • Pegué código real (sin datos de producción ni credenciales) y pedí análisis
  • Pedí la refactorización y guardé el resultado
  • Pregunté por el patrón de diseño aplicable
  • Comparé la versión original vs refactorizada
💡
Sobre cultura (tu bloqueo): La IA no reemplaza al programador — es un par de ojos extra. Tú decides qué aceptar y qué descartar. El código que produce Hermes siempre debe ser revisado y probado antes de integrarse.
📎 Evidencia del Día 3
  • Código original vs código refactorizado por Hermes
  • Explicación del patrón aplicado
Día 4 · Lección 2 (continuación)
SOUL.md — tu agente de desarrollo personalizado
30 min · Hermes Agent + editor de texto
Concepto clave
El SOUL.md configura permanentemente quién es tu agente. Para Desarrollo, esto significa: lenguajes que usas, patrones que prefieres, convenciones del equipo y restricciones de datos.

Crear tu SOUL.md de Desarrollo

# SOUL — Agente de Desarrollo UJMD

Eres un asistente de desarrollo de software de la Dirección de
Servicios Informáticos de la UJMD, El Salvador.

## Stack tecnológico del área
- Lenguajes principales: Java, [agrega los tuyos]
- Bases de datos: [MySQL / Oracle / etc.]
- Frameworks: [Spring / etc.]

## Tu rol
- Revisas código y señalas bugs, duplicaciones y mejoras
- Generas documentación técnica en español
- Ayudas a estructurar requerimientos funcionales
- Propones arquitectura y patrones de diseño

## Reglas de código
- Nunca uses datos reales de estudiantes, empleados ni sistemas
- Usa siempre datos ficticios en los ejemplos
- Si ves credenciales en el código, advierte inmediatamente

## Estilo
Técnico pero claro. Explica el "por qué" de cada sugerencia.

Actividades del Día 4

  • Creé mi SOUL.md con el stack real del área de Desarrollo
  • Verifiqué que Hermes lo reconoce (hermes chat -q "¿cuál es tu rol y stack?")
  • Probé un prompt de código con el SOUL.md activo vs sin él
  • Ajusté el SOUL.md con el lenguaje y patrones que realmente usa el área
📎 Evidencia del Día 4
  • Contenido de tu SOUL.md (copia el texto)
  • Respuesta de Hermes al preguntarle su stack y rol
Día 5 · Caso Ancla
Revisión real del módulo de Graduados con Hermes
60 min · Hermes Agent (con SOUL.md activo)
🎯
Tu caso ancla: Revisar y refactorizar código legado con IA como pair programmer. Usarás el módulo de Graduados que ya conoces.
⚠️
Antes de empezar: Solo pega fragmentos de código (funciones, clases). No pegues cadenas de conexión, credenciales, tokens ni datos reales de graduados. Usa solo la lógica de negocio.

Flujo de trabajo con Hermes para el módulo

# Paso 1: Análisis inicial
hermes chat -q "Analiza este fragmento del módulo de Graduados e identifica: bugs potenciales, código duplicado y áreas de mejora. [pega el fragmento]"

# Paso 2: Refactorización
hermes chat -q "Ahora refactoriza este fragmento aplicando las mejoras que identificaste. Mantén la misma funcionalidad."

# Paso 3: Documentación automática
hermes chat -q "Genera la documentación técnica en español para el fragmento refactorizado. Formato: JavaDoc."

# Paso 4: Pruebas sugeridas
hermes chat -q "¿Qué casos de prueba debería escribir para verificar que la refactorización no rompió nada?"

Actividades del Día 5

  • Seleccioné un fragmento real del módulo de Graduados (sin datos sensibles)
  • Completé los 4 pasos del flujo con Hermes
  • Revisé y validé la refactorización sugerida
  • Guardé versión original + refactorizada + documentación generada
  • Anoté qué parte de la sugerencia acepté y qué descarté (y por qué)
📎 Evidencia del Día 5
  • Fragmento original vs refactorizado (diff)
  • Documentación JavaDoc generada
  • Lista de casos de prueba sugeridos
